JA3 Fingerprinting Lab

Laboratorio interactivo para entender cómo se generan las huellas TLS JA3, cómo identificar clientes maliciosos y cómo integrar esta técnica en el flujo SOC.

JA3 (John Althouse, Jeff Atkinson, Josh Atkins — Salesforce, 2017) extrae cinco campos del mensaje TLS Client Hello y calcula su hash MD5. El resultado es una huella de 32 chars que identifica la implementación TLS del cliente — independientemente de la IP o el dominio contactado.

Fórmula JA3

JA3 construye un string con los 5 campos, separando los valores de cada lista con - y los campos entre sí con ,. Luego calcula el MD5 del resultado:

FÓRMULA Concatenación de campos 
JA3_STRING = SSLVersion,Ciphers,Extensions,EllipticCurves,ECPointFormats

Ejemplo pedagógico (TLS 1.2):
  SSLVersion  = 771  (TLS 1.2 = 0x0303)
  Ciphers     = 49199-49195-158-60
                  0xC02F = TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
                  0xC02B = TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256
                  0x009E = TLS_DHE_RSA_WITH_AES_128_GCM_SHA256
                  0x003C = TLS_RSA_WITH_AES_128_CBC_SHA256
  Extensions  = 0-23-65281-10-11-16-5-13
                  SNI, extended_master_secret, renegotiation_info,
                  supported_groups, ec_point_formats, ALPN,
                  status_request, signature_algorithms
  Curves      = 29-23-24  (x25519, secp256r1, secp384r1)
  ECFormats   = 0

JA3_STRING = "771,49199-49195-158-60,0-23-65281-10-11-16-5-13,29-23-24,0"

JA3_HASH   = MD5(JA3_STRING)
           = ada70206e40642a3e4461f35503241d5  ← ejemplo calculado
             # Usá la calculadora interactiva para verificar el hash real
¿Por qué MD5? JA3 usa MD5 por velocidad de cálculo en línea de red — no por seguridad criptográfica. Con ~10 mil millones de hashes/segundo en GPU, el análisis en tiempo real de tráfico no introduce latencia notable.

Valores GREASE — RFC 8701

Los browsers modernos insertan valores GREASE (Generate Random Extensions And Sustain Extensibility) para garantizar que los servidores ignoren valores TLS desconocidos. JA3 los filtra antes de concatenar para mantener huellas consistentes entre sesiones del mismo cliente.

Valores GREASE (decimal) Hex ¿Dónde aparece?
2570, 6682, 10794, 14906, 19018, 23130, 27242, 31354 0x0a0a…0x7a7a Ciphers, Extensions, Curves
35466, 39578, 43690, 47802, 51914, 56026, 60138, 64250 0x8a8a…0xfafa Ciphers, Extensions, Curves

Flujo SOC con JA3. Desde la alerta inicial hasta el cierre del incidente. Los pasos asumen que ya tenés visibilidad TLS (proxy, IDS, EDR o tap de red con exportación a SIEM).

Flujo de detección y respuesta

1
🔔 Alerta SIEM / IDS — hash JA3 sospechoso

El SIEM correlaciona un hash JA3 contra la lista de firmas conocidas. Puede ser una regla Snort, una alerta de QRadar o un indicador de FortiGate. La alerta incluye: src_ip, dst_ip, dst_port, ja3_hash, timestamp.

2
🔍 Enriquecimiento del indicador

Verificar el hash en feeds externos: sslbl.abuse.ch, ja3er.com, VirusTotal (Network tab). Correlacionar con el historial del host origen: ¿es un servidor, una estación de trabajo, un dispositivo IoT? ¿Cuándo fue la primera vez que viste este JA3 en tu red?

3
📦 Captura de evidencia

Si confirmás que es malicioso: capturar el PCAP completo del flujo, exportar logs del SIEM del host afectado (Event IDs 3, 7045, 4688 en Windows), tomar snapshot de memoria si el host está activo. El PCAP puede revelar el payload C2.

4
🔗 Análisis de pivote

Buscar el mismo JA3 hash en toda la red: ¿hay otros hosts con la misma huella contactando el mismo C2? El mismo JA3 en múltiples fuentes sugiere infección masiva o movimiento lateral. Buscar también por dst_ip y certificate_hash.

5
🛑 Contención y bloqueo

Bloquear la IP/dominio de destino en el firewall y proxy. Si el malware usa DGA, el bloqueo por IP es insuficiente — evaluar aislamiento del host. Agregar el JA3 hash a la lista de bloqueo en Snort/Suricata, QRadar y FortiGate.

6
📝 Documentación y lecciones aprendidas

Actualizar el playbook con el nuevo JA3. Agregar el hash a la watchlist del SIEM. Reportar el indicador a plataformas de threat intel compartido (MISP, ISACs). Evaluar si la detección fue oportuna o tardía y ajustar el umbral de alerta.

JA3 vs JA3S. JA3 identifica al cliente TLS por su Client Hello. JA3S (JA3 Server) identifica al servidor por su Server Hello. Juntos permiten fingerprinting bidireccional de una sesión TLS completa — fundamental para detección de infraestructura C2.

¿Qué captura cada uno?

JA3 es el fingerprint del CLIENT HELLO: identifica al cliente, la herramienta o el malware que inicia la conexión. Los campos capturados son los que el cliente ofrece al servidor — su lista de ciphers preferidos, extensiones soportadas, curvas, etc.

JA3S es el fingerprint del SERVER HELLO: identifica al servidor o la infraestructura que responde. La fórmula es más simple porque el servidor elige un solo cipher de la lista del cliente:

FÓRMULA JA3S Más simple que JA3 — el servidor elige un único cipher 
JA3S_STRING = SSLVersion,Cipher,Extensions
              └── SSLVersion : versión negociada (elegida por el servidor)
              └── Cipher     : UNA cipher suite (la elegida, no una lista)
              └── Extensions : extensiones presentes en el Server Hello

Ejemplo:
  JA3S_STRING = "771,49199,0-65281"
  JA3S_HASH   = MD5("771,49199,0-65281")

Extensions Lista de extensiones del cliente Extensiones presentes en el Server Hello

Caso SOC real: detección de infraestructura C2

La correlación JA3 + JA3S es más poderosa que JA3 solo. Un atacante puede cambiar su JA3 randomizando el cipher order, pero no puede cambiar el JA3S del servidor C2. Si ves el mismo JA3S respondiendo a múltiples JA3s sospechosos, tenés infraestructura C2 identificada — aunque el cliente cambie su huella entre conexiones.

Ejemplo de par documentado — Cobalt Strike:

Dirección Hash Mensaje TLS Fuente
ORIGEN_IP → DEST_IP:443 JA3: de350869b8c85de67a350c8d186f11e6 Client Hello Fox-IT Research
DEST_IP:443 → ORIGEN_IP JA3S: ae4edc6faf64d08308082ad26be60767 Server Hello Fox-IT Research

Workflow de detección en 3 pasos

1
🔔 Detectar JA3 sospechoso en Client Hello

El SIEM o IDS genera una alerta cuando detecta un JA3 hash en la lista de amenazas conocidas. La alerta incluye la IP origen, destino, puerto y timestamp. También puede detectar JA3s anómalos por baja entropía o ciphers inusuales.

2
🔍 Capturar JA3S del Server Hello correspondiente

Del mismo flujo TCP, extraer el Server Hello y calcular el JA3S. Herramientas como Zeek lo hacen automáticamente y lo exportan junto al JA3 en el mismo log (ssl.log). En Wireshark: filtrar por el mismo stream y buscar el Server Hello.

3
🔗 Correlacionar par JA3 + JA3S contra threat intel

Verificar el par JA3 + JA3S en bases de threat intel. El JA3S es especialmente útil para pivot: buscar todas las IPs que hayan recibido el mismo JA3S — eso identifica el servidor C2 aunque el cliente cambie su JA3. Documentar el par en el MISP del SOC para compartir con la comunidad.

Limitaciones de JA3. JA3 es una herramienta poderosa pero no infalible. Conocer sus limitaciones permite usar el indicador correctamente: como disparador de investigación, no como evidencia concluyente.

Timeline — evolución de las técnicas de evasión

2017
JA3 publicado
John Althouse, Jeff Atkinson y Josh Atkins (Salesforce) publican el algoritmo JA3. Inmediatamente útil para identificar herramientas de ataque como Metasploit y Cobalt Strike.
2018
uTLS aparece
La librería Go uTLS permite imitar fingerprints de browsers (Chrome, Firefox) a nivel TLS. Cualquier herramienta escrita en Go puede pasar por un browser legítimo a nivel JA3.
2019
JA3 Randomization
Herramientas como curl agregan opciones para randomizar el cipher order. Resultado: JA3 diferente en cada conexión del mismo cliente. Algunos malware modernos adoptan esta técnica automáticamente.
2020
TLS 1.3 — adopción masiva
TLS 1.3 cifra muchas extensiones en el handshake. JA3 pierde información útil. La visibilidad es mayor con TLS 1.2 y sesiones que no usan 0-RTT.
2021
Malware imita Chrome/Firefox
RATs y frameworks C2 modernos (Sliver, Brute Ratel) implementan fingerprints de Chrome o Firefox por defecto. La huella JA3 parece legítima sin serlo.
2023+
QUIC / HTTP/3
HTTP/3 usa QUIC sobre UDP en lugar de TCP+TLS. JA3 no aplica a QUIC — requiere técnicas de fingerprinting distintas (QUIC fingerprinting, JARM adaptado). Tendencia creciente.

Limitaciones en detalle

⚠ LIMITACIÓN 1 — JA3 Randomization

Herramientas y malware modernos pueden randomizar el orden de los cipher suites en cada conexión. Resultado: el hash JA3 cambia en cada sesión del mismo cliente, haciendo inútil la comparación directa con firmas conocidas.

BASH curl con cipher order personalizado 
# curl puede especificar cipher order diferente
curl --tls-cipher-list 'DEFAULT:!SHA' https://example.com
# Resultado: JA3 diferente al default de curl
# Algunos malware hacen esto automáticamente en cada beacon

Impacto SOC: No podés depender de un único hash JA3 para identificar una herramienta si usa randomización. Correlacionar con IP, timing, volumen de conexiones y otros indicadores.

⚠ LIMITACIÓN 2 — uTLS — Librería de impersonación

La librería uTLS (Go) permite que cualquier aplicación imite exactamente el fingerprint TLS de un browser. Un agente malicioso escrito en Go puede presentar el JA3 de Chrome y pasar desapercibido a nivel de fingerprinting TLS.

GO uTLS — impersonación de Chrome 
// Go code que imita Chrome TLS fingerprint exactamente
tlsConfig := &utls.Config{ServerName: "example.com"}
conn := utls.UClient(tcpConn, tlsConfig, utls.HelloChrome_Auto)
// Resultado: JA3 idéntico al de Chrome — indistinguible por hash

Detección: Buscar inconsistencias — JA3 de Chrome pero sin User-Agent header de Chrome, sin cookies de sesión típicas, patrones de timing inusuales, o acceso a paths que un browser legítimo no visitaría.

⚠ LIMITACIÓN 3 — TLS 1.3 y extensiones cifradas

En TLS 1.3 muchas extensiones van cifradas en el handshake (Encrypted Client Hello — ECH). JA3 pierde información significativa porque no puede leer esas extensiones. Por eso JA3 funciona mejor con TLS 1.2 y sesiones que no usan 0-RTT (resumption).

Alternativa complementaria: Usar JARM (fingerprinting activo del servidor) que funciona independientemente de la versión TLS del cliente. JARM envía varios Client Hellos especialmente crafteados y analiza las respuestas del servidor.

⚠ LIMITACIÓN 4 — Colisiones JA3

Múltiples aplicaciones pueden generar el mismo hash JA3. La probabilidad es baja pero real: una librería TLS puede producir la misma huella que un malware si usa la misma versión y configuración por defecto. MD5 fue elegido por velocidad, no por ausencia de colisiones.

Regla de oro SOC: Nunca bloquear un host solo por JA3. Siempre correlacionar con contexto: IP de destino, certificado, comportamiento del host, hora, volumen.

JA3 es un indicador, no una prueba. En un SOC maduro, un match de JA3 sospechoso dispara una investigación, no un bloqueo automático. La potencia de JA3 está en la correlación: mismo JA3 en múltiples hosts, par JA3+JA3S consistente, JA3 anómalo combinado con comportamiento de beaconing — eso es lo que construye un caso sólido.
