Análisis TLS con Zeek

Zeek genera logs estructurados del tráfico de red — incluyendo ssl.log con fingerprints JA3/JA3S. Aprenderás a huntar amenazas TLS usando Zeek en un entorno SOC real.

🔐 JA3 Core 🦈 Wireshark 🐷 Snort/Suricata 📊 QRadar 🛡️ FortiGate 🦎 Zeek

Zeek (antes Bro) es un framework de análisis de tráfico de red pasivo. No bloquea tráfico, lo analiza y genera logs estructurados. Es el estándar en análisis forense de red en SOC.

Arquitectura simplificada

Captura de paquetes
→
Zeek Engine
→
Analyzers
→
ssl.log · conn.log · dns.log · http.log…

¿Por qué Zeek para TLS/JA3?

  • Genera ssl.log automáticamente con campos TLS completos: cipher, version, ja3, ja3s, SNI
  • No requiere instrumentación adicional — funciona sobre tráfico existente (tap o SPAN port)
  • Ideal para análisis forense post-incident y hunting proactivo
  • Zeek 5.x+ incluye JA3/JA3S de forma nativa en ssl.log

Instalación (referencia)

BASH Zeek 5.x en Ubuntu/Debian 
# Zeek 5.x en Ubuntu/Debian
sudo apt-get install zeek

# Agregar JA3 plugin (Zeek < 5.0)
zkg install zeek/salesforce/ja3

# Zeek 5.0+ incluye JA3 nativo
# Verificar versión instalada

"En un SOC maduro, Zeek corre sobre un tap de red o SPAN port, ingiriendo todo el tráfico. Los logs se envían a un SIEM (QRadar, Splunk, Elastic) para correlación. El ssl.log es gold para análisis TLS."

ssl.log es el log principal de Zeek para análisis TLS. Cada entrada representa una sesión TLS completa con todos sus metadatos, incluyendo los fingerprints JA3 y JA3S.

Campos clave del ssl.log

Campo Descripción Relevancia SOC
ts Timestamp Unix Cronología de ataque
uid Connection ID único Pivot entre logs
id.orig_h / id.resp_h IPs origen/destino Identificación de endpoints
version Versión TLS TLS 1.0/1.1 = red flag
cipher Cipher negociado Ciphers débiles = riesgo
server_name SNI del cliente Infraestructura C2 sin SNI = sospechoso
ja3 Fingerprint cliente Tool/malware identification
ja3s Fingerprint servidor C2 infrastructure fingerprint
resumed Sesión reanudada Beaconing pattern
established Handshake completo Conexión exitosa

Comandos Zeek para captura

BASH Captura en vivo y análisis de PCAP 
# Captura en vivo
zeek -i eth0 local.zeek

# Análisis de PCAP (con JA3 habilitado en versiones anteriores a 5.0)
zeek -r traffic.pcap /opt/zeek/share/zeek/site/ja3/ja3.zeek

# Ver ssl.log en tiempo real
tail -f /opt/zeek/logs/current/ssl.log | zeek-cut ts id.orig_h id.resp_h ja3 server_name

# Habilitar JA3 en site/local.zeek (Zeek < 5.0)
echo '@load protocols/ssl/ja3' >> /opt/zeek/share/zeek/site/local.zeek

Formato ja3.log (versiones antiguas de Zeek)

ZEEK LOG FORMAT ja3.log — presente en Zeek anterior a 5.x 
#fields ts  uid  id.orig_h  id.orig_p  id.resp_h  id.resp_p  ja3  ja3s
En Zeek 5.x+ los campos ja3 y ja3s están integrados directamente en ssl.log. El archivo separado ja3.log solo aparece en instalaciones antiguas con el plugin de Salesforce cargado manualmente.

zeek-cut es la navaja suiza para analizar logs de Zeek desde CLI. Permite extraer columnas específicas del formato TSV de Zeek de manera eficiente, similar a awk pero con conocimiento nativo del formato de logs.

Query 1 — Top JA3 en la red

BASH · zeek-cut Top 20 fingerprints más frecuentes 
# Top 20 fingerprints más frecuentes
cat ssl.log | zeek-cut ja3 | sort | uniq -c | sort -rn | head 20

Query 2 — Buscar JA3 malicioso conocido

BASH · zeek-cut Búsqueda puntual y búsqueda por lista 
# Buscar Cobalt Strike JA3
cat ssl.log | zeek-cut ts uid id.orig_h id.resp_h id.resp_p ja3 server_name | \
  grep "de350869b8c85de67a350c8d186f11e6"

# Buscar cualquier JA3 de una lista (bash)
MALICIOUS_JA3=(
  "de350869b8c85de67a350c8d186f11e6"  # Cobalt Strike
  "e7d705a3286e19ea42f587b344ee6865"  # Metasploit
  "8bab54f22de07d88e90f5e8e5edb5a69"  # Emotet
)
for ja3 in "${MALICIOUS_JA3[@]}"; do
  echo "=== Buscando: $ja3 ==="
  grep "$ja3" ssl.log | zeek-cut ts id.orig_h id.resp_h server_name
done

Query 3 — Correlación JA3 + JA3S (par C2)

BASH · zeek-cut Pares únicos cliente+servidor para fingerprint completo de sesión 
# Encontrar pares JA3+JA3S únicos
cat ssl.log | zeek-cut id.orig_h id.resp_h ja3 ja3s | \
  awk '{if($3!="" && $4!="") print $3"---"$4}' | \
  sort | uniq -c | sort -rn

# Pivot: buscar todos los clientes que contactaron mismo JA3S (misma infraestructura C2)
cat ssl.log | zeek-cut id.orig_h id.resp_h ja3s | \
  awk '/ae4edc6f/ {print $1, $2}' | sort -u

Query 4 — Detección de Beaconing

BASH + PYTHON Calcular intervalos y detectar C2 automatizado 
# Paso 1 — Extraer y ordenar por host+destino
cat ssl.log | zeek-cut ts id.orig_h id.resp_h ja3 | \
  awk 'BEGIN{OFS="\t"}{print $1,$2,$3,$4}' | \
  sort -k2,3 > /tmp/sorted_ssl.txt

# Paso 2 — Python helper para detectar beaconing
python3 << 'EOF'
import sys
from collections import defaultdict

connections = defaultdict(list)
with open('/tmp/sorted_ssl.txt') as f:
    for line in f:
        ts, src, dst, ja3 = line.strip().split('\t')
        connections[(src, dst, ja3)].append(float(ts))

for key, timestamps in connections.items():
    if len(timestamps) < 5:
        continue
    timestamps.sort()
    intervals = [timestamps[i+1]-timestamps[i] for i in range(len(timestamps)-1)]
    avg = sum(intervals)/len(intervals)
    variance = sum((x-avg)**2 for x in intervals)/len(intervals)
    # Beacon si intervalo regular (variance baja) y promedio ~5min
    if variance < 1000 and 240 < avg < 360:
        src, dst, ja3 = key
        print(f"⚠️  BEACON DETECTADO: {src} → {dst} | JA3: {ja3} | Intervalo: {avg:.0f}s")
EOF

Query 5 — Anomalías TLS

BASH · zeek-cut Puertos inusuales, sin SNI, versiones obsoletas 
# Conexiones TLS en puertos no estándar
cat ssl.log | zeek-cut id.orig_h id.resp_h id.resp_p ja3 server_name | \
  awk '$3 != 443 && $3 != 8443 && $3 != 993 && $3 != 465'

# Sin SNI (posible C2 directo por IP)
cat ssl.log | zeek-cut ts id.orig_h id.resp_h server_name ja3 | \
  awk '$4 == "-"' | head 20

# TLS versiones obsoletas
cat ssl.log | zeek-cut id.orig_h id.resp_h version | \
  grep -E 'TLSv10|TLSv11|SSLv'

Script Zeek de alerta automática

ZEEK SCRIPT /opt/zeek/site/ja3_alert.zeek — detección nativa 
# /opt/zeek/site/ja3_alert.zeek
# Cargarlo con: zeek -i eth0 ja3_alert.zeek

module JA3Alert;

export {
  # Hashes JA3 maliciosos conocidos
  const malicious_ja3: set[string] = {
    "de350869b8c85de67a350c8d186f11e6",   # Cobalt Strike
    "e7d705a3286e19ea42f587b344ee6865",   # Metasploit
    "8bab54f22de07d88e90f5e8e5edb5a69",   # Emotet
  } &redef;

  redef enum Notice::Type += { MaliciousJA3 };
}

event ssl_client_hello(c: connection, version: count, record_version: count,
                       possible_ts: time, client_random: string,
                       session_id: string, ciphers: index_vec,
                       comp_methods: index_vec)
{
  # JA3 es calculado internamente por Zeek si está habilitado
  # Este hook se ejecuta cuando hay coincidencia en ssl.log
  if ( c$ssl?$ja3 && c$ssl$ja3 in malicious_ja3 ) {
    NOTICE([$note=JA3Alert::MaliciousJA3,
            $conn=c,
            $msg=fmt("JA3 malicioso detectado: %s desde %s hacia %s:%s",
                     c$ssl$ja3, c$id$orig_h, c$id$resp_h, c$id$resp_p),
            $identifier=cat(c$id$orig_h, c$ssl$ja3)]);
  }
}

Flujo de trabajo SOC con Zeek para investigar tráfico TLS sospechoso. Seis pasos ordenados desde la captura hasta la escalación, con comandos exactos listos para usar en producción.

Workflow de análisis con Zeek

①
Captura — Iniciar Zeek sobre interfaz o PCAP

Iniciar Zeek en modo vivo o sobre un archivo capturado:
zeek -i eth0 local.zeek — captura en tiempo real
zeek -r archivo.pcap — análisis post-incident de PCAP existente
Los logs se generan en el directorio actual o en /opt/zeek/logs/current/.

②
Review inicial — Leer ssl.log buscando anomalías

tail -f ssl.log | zeek-cut ts id.orig_h id.resp_h ja3 server_name
Buscar señales de alerta temprana:
— Sin SNI (server_name == "-"): posible C2 directo por IP
— Puertos no estándar: 4443, 8080, 8888, etc.
— Conexiones muy frecuentes al mismo destino: patrón beacon

③
Extracción JA3 — Top hashes y comparación con threat intel

cat ssl.log | zeek-cut ja3 | sort | uniq -c | sort -rn
Comparar hashes extraídos contra feeds de threat intel:
— sslbl.abuse.ch: base de datos JA3 maliciosos actualizada
— ja3er.com: fingerprints por aplicación conocida
Un hash desconocido con alta frecuencia merece investigación inmediata.

④
Correlación JA3+JA3S — Identificar infraestructura C2

Los pares JA3+JA3S únicos generan un fingerprint completo de la sesión TLS.
Pivot por JA3S para encontrar múltiples víctimas del mismo C2:
cat ssl.log | zeek-cut id.orig_h ja3s | awk '/HASH_JA3S/ {print $1}' | sort -u
Si múltiples IPs internas aparecen con el mismo JA3S, es un indicador fuerte de compromiso lateral.

⑤
Análisis temporal — Detección de beaconing

Los C2 automatizados se comunican a intervalos regulares (beaconing).
Calcular la varianza de los intervalos entre conexiones al mismo destino.
Varianza baja + intervalo promedio constante = alta probabilidad de C2 automatizado.
El script Python de la tab "Hunting" implementa esta lógica lista para usar.

⑥
Documentación y escalación

Exportar evidencia con campos completos para el ticket de incidente:
cat ssl.log | zeek-cut ts uid id.orig_h id.resp_h id.resp_p ja3 ja3s server_name > evidence.tsv
Correlacionar con otros logs de Zeek: conn.log (duración, bytes), dns.log (dominios resueltos), http.log (requests HTTP).
Escalar con: IPs IOC, hashes JA3/JA3S, timestamps exactos, UIDs para pivot en SIEM.
