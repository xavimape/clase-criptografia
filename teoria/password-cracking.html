08 — Password Cracking y John the Ripper

Conexión pedagógica: Este módulo integra los conceptos de hashing, fundamentos criptográficos y análisis de amenazas aplicados a la auditoría de contraseñas y detección de ataques en entornos SOC.

1. ¿Qué es Password Cracking?

El password cracking es el proceso de recuperar contraseñas en texto plano a partir de sus representaciones almacenadas (generalmente hashes). No es lo mismo que romper un algoritmo criptográfico.

Cracking de contraseñas vs. criptoanálisis

Concepto Objetivo Ejemplo
Password cracking Recuperar el plaintext desde un hash MD5 de "admin" → "admin"
Criptoanálisis Encontrar debilidades en el algoritmo Colisión MD5 (Wang et al., 2004)
Fuerza bruta Probar todas las combinaciones posibles PIN de 4 dígitos: 10.000 intentos
Ataque de diccionario Probar una lista de palabras conocidas rockyou.txt contra hash NTLM

El punto clave: el cracking no explota una vulnerabilidad matemática en SHA256 o bcrypt. Explota contraseñas débiles y algoritmos mal elegidos para almacenarlas.

¿Por qué importa en SOC?

  • Las brechas de datos incluyen frecuentemente bases de hashes de contraseñas
  • Los atacantes crackean esos hashes para reutilizar credenciales (credential stuffing)
  • Un analista debe evaluar el riesgo de una filtración según el algoritmo de hash usado
  • La detección de ataques de brute force es una alerta frecuente en SIEM

2. John the Ripper

Historia y uso legítimo

John the Ripper (JtR) fue creado por Solar Designer en 1997 y es hoy uno de los auditores de contraseñas más usados en pentesting, forensia y cumplimiento normativo. Soporta cientos de formatos de hash, desde MD5 hasta bcrypt y hashes de sistemas Windows (NTLM).

Caso de uso Descripción ¿Requiere autorización?
Auditoría de política de passwords Verificar que los usuarios no usen contraseñas débiles Sí — del propietario del sistema
Forensia digital Recuperar acceso a dispositivos o archivos cifrados Sí — orden judicial o dueño
Pentesting (Red Team) Simular ataques reales en alcance definido Sí — contrato / Rules of Engagement
CTF / laboratorio Aprendizaje en entornos controlados No aplica (ambiente propio)
⚖️ Marco legal: En Argentina, la Ley 26.388 penaliza el acceso ilegítimo a sistemas informáticos. En EEUU aplica el CFAA. En Europa, la Directiva NIS2. El uso de JtR sin autorización explícita constituye un delito en la mayoría de jurisdicciones.

Modos de operación

# Ataque de diccionario
john hashes.txt --wordlist=rockyou.txt --format=raw-md5

# Fuerza bruta incremental
john hashes.txt --incremental

# Con reglas de mutación (combina ambos enfoques)
john hashes.txt --wordlist=rockyou.txt --rules=jumbo

# Ver resultados crackeados
john --show hashes.txt

3. Ataque de Diccionario

¿Cómo funciona?

El atacante toma una lista de contraseñas conocidas (wordlist), hashea cada una con el mismo algoritmo que usó el sistema, y compara con el hash objetivo. Si hay coincidencia, la contraseña queda revelada.

Para cada palabra en rockyou.txt:
    hash_calculado = MD5(palabra)
    if hash_calculado == hash_objetivo:
        FOUND: contraseña = palabra
        break

rockyou.txt — la wordlist más usada

En 2009, la empresa RockYou.com sufrió una brecha que expuso 14.344.391 contraseñas en texto plano. Esta lista se convirtió en el estándar de facto para ataques de diccionario. En tests reales, permite crackear entre el 60-80% de los hashes MD5 del mundo empresarial en minutos.

Wordlist Contraseñas Origen Uso
rockyou.txt 14.3 millones Brecha RockYou 2009 Estándar en pentesting
Have I Been Pwned ~12 mil millones Múltiples brechas Verificación de exposure
CrackStation 1.5 mil millones Diccionarios + brechas Auditorías exhaustivas
SecLists Variable Curado por comunidad Auditorías especializadas

Variante: Password Spraying

En lugar de probar muchas contraseñas contra una cuenta, el atacante prueba una contraseña común contra muchas cuentas. Evita los lockouts por cuenta individual. Ejemplo: probar "Empresa2024!" contra todos los usuarios del Active Directory.

4. Fuerza Bruta

¿Cómo funciona?

Prueba sistemáticamente todas las combinaciones posibles dentro de un charset y longitud definidos. No requiere conocimiento previo de la contraseña. El espacio de búsqueda crece exponencialmente con la longitud.

Combinaciones = tamaño_charset ^ longitud

Ejemplos:
  PIN 4 dígitos      → 10⁴     = 10,000           combinaciones
  8 chars a-z        → 26⁸     = 208,827,064,576   combinaciones
  8 chars a-zA-Z0-9  → 62⁸     = 218,340,105,584,896
  12 chars full ASCII → 95¹²   = 540,360,087,662,636,963,840

Velocidades reales (GPU RTX 3090, 2024)

Algoritmo Hashes/seg 8 chars alfanum. (peor caso)
MD5 ~10,000,000,000 ~6 horas
SHA1 ~3,300,000,000 ~18 horas
SHA256 ~1,200,000,000 ~2 días
bcrypt (cost 12) ~15,000 ~462 años
Argon2id ~800 > 8,000 años

Estrategias de optimización que usa el atacante

  • Maskattack: patrón conocido (ej: "Empresa" + 4 dígitos)
  • Reglas de mutación: "password" → "P@ssw0rd!", "password1", "PASSWORD"
  • GPU clusters: múltiples GPUs en paralelo multiplicando la velocidad
  • Rainbow tables: tablas pre-computadas (ver siguiente sección)

5. Rainbow Tables

Concepto

Una rainbow table es una estructura de datos que almacena pares hash → plaintext pre-computados para un charset y longitud dados. El atacante solo necesita buscar en la tabla en lugar de calcular en tiempo real. Es un tradeoff tiempo vs. memoria: el cómputo se hace una vez, la búsqueda es instantánea.

Creación (una vez, cara en tiempo):
  Pre-compute MD5("a"), MD5("b"), ..., MD5("zzzzzzzz")
  Almacenar tabla: {hash: plaintext, ...}

Uso (instantáneo):
  lookup(tabla, hash_objetivo) → plaintext

Mitigación: el salt

Un salt es un valor aleatorio único agregado a la contraseña antes de hashearla. Invalida completamente las rainbow tables porque el atacante tendría que generar una tabla por cada salt posible (inviable).

# Sin salt (vulnerable a rainbow tables):
hash = MD5("password")         → siempre 5f4dcc3b...

# Con salt (inmune a rainbow tables):
salt = "a3f8b2"
hash = MD5(salt + "password")  → hash único
hash = MD5(salt + "password")  → mismo hash (verificación OK)
# Pero salt diferente por usuario → hash diferente para la misma contraseña

bcrypt y Argon2 incluyen el salt automáticamente en el output. Dos usuarios con la misma contraseña tendrán hashes completamente distintos.

6. Hashes débiles vs. algoritmos seguros para passwords

Por qué MD5 y SHA1 son inseguros para contraseñas

MD5 y SHA1 no fueron diseñados para almacenar contraseñas. Fueron creados para verificación de integridad de archivos, donde la velocidad es una virtud. Para contraseñas, esa misma velocidad es una vulnerabilidad fatal.

Algoritmo Año Diseñado para GPU hashes/seg Salt automático Para passwords
MD5 1992 Checksums ~10B No ❌ Nunca
SHA1 1995 Integridad / firmas ~3.3B No ❌ Nunca
SHA256 2001 TLS / firmas / integridad ~1.2B No ⚠️ No ideal
bcrypt 1999 Almacenar passwords ~15K Sí ✅ Sí
Argon2id 2015 Almacenar passwords ~800 Sí ⭐ Óptimo

bcrypt en detalle

bcrypt incorpora un work factor (o cost factor) que determina cuántas iteraciones se realizan. A mayor cost factor, más lento y más seguro. La recomendación actual de OWASP es cost factor 12 (≈ 150ms en servidor moderno).

# Output real de bcrypt (contiene todo lo necesario para verificar):
$2b$12$EixZaYVK1fsbw1ZfbX3OXePaWxn96p36WQoeG6Lruj3vjPGga31lW
 │   │  │──────────────────────────────────────────────────────│
 │   │  22 chars de salt + 31 chars de hash
 │   └─ cost factor (2¹² = 4096 iteraciones)
 └─ versión de bcrypt

Argon2id — el estado del arte

Ganador del Password Hashing Competition (PHC) en 2015. Es memory-hard: requiere grandes cantidades de RAM además de CPU, haciendo que los ataques con GPUs/ASICs sean extremadamente costosos.

$argon2id$v=19$m=65536,t=3,p=4$[salt]$[hash]
                │         │    │
                │         │    └─ 4 threads en paralelo
                │         └─ 3 iteraciones
                └─ 65536 KB (64MB) de memoria requerida

7. Mitigación

Para desarrolladores (almacenamiento seguro)

  • Usar bcrypt (cost ≥ 12) o Argon2id — nunca MD5, SHA1 o SHA256 desnudos
  • Salt automático — bcrypt/Argon2 lo generan internamente; no implementar salt manual con SHA256
  • Work factor adaptativo — aumentar el cost factor a medida que el hardware mejora
  • No almacenar passwords en texto plano jamás — parece obvio, pero ocurre
  • Pepper — secreto a nivel servidor (no en BD) para un factor adicional

Para usuarios (higiene de contraseñas)

  • 16+ caracteres con mezcla de tipos: imposible por fuerza bruta incluso con MD5
  • Contraseña única por servicio: credential stuffing explota reutilización
  • Gestor de contraseñas: Bitwarden, 1Password, KeePass — genera y almacena secuencias aleatorias
  • MFA: mitigación más efectiva — bloquea el acceso incluso con la contraseña comprometida

Para organizaciones (controles técnicos)

  • Account lockout policy: bloqueo tras N intentos fallidos (ej: 10 intentos → 30 min)
  • Rate limiting en endpoints de autenticación
  • CAPTCHA / bot detection en formularios de login
  • Have I Been Pwned API: bloquear passwords que aparecen en brechas conocidas
  • Auditorías periódicas con JtR/Hashcat para detectar contraseñas débiles proactivamente

8. Relación con SOC

Detección de brute force en SIEM

# Lógica de alerta — SIEM genérico
event_type = "failed_login"
AND count(event) > 10
AND timeframe = "60 seconds"
AND source_ip = same

# Windows Event IDs a monitorear:
4625 → An account failed to log on
4740 → A user account was locked out
4648 → A logon was attempted using explicit credentials
4771 → Kerberos pre-authentication failed (AD)

# Linux /var/log/auth.log:
"Failed password for [user] from [IP] port [N]"
"Invalid user [user] from [IP]"
"authentication failure; logname= uid=0"

Credential Stuffing

El atacante usa pares usuario:contraseña de brechas anteriores (recopiladas de múltiples filtraciones) y los prueba de forma distribuida contra el objetivo. Diferente al brute force: cada IP intenta pocas veces, evitando lockouts por IP.

# Indicadores de credential stuffing:
- Muchas IPs distintas con 1-2 intentos fallidos cada una
- Usuarios válidos con intentos fallidos distribuidos en el tiempo
- Picos de tráfico en el endpoint de login desde múltiples ASNs
- Correlación con brechas recientes en Have I Been Pwned

Pass-the-Hash (PtH)

En entornos Windows, el atacante puede autenticarse usando el hash NTLM directamente, sin necesidad de crackearlo. Esto ocurre en movimiento lateral post-compromiso. El hash actúa como la contraseña misma en el protocolo NTLM.

# Detección de PtH en Windows:
Event ID 4624, Logon Type 3 (Network)
con campo "Authentication Package" = NTLM
desde un host no esperado para ese usuario

Evaluación de riesgo en una brecha con hashes

Algoritmo en la brecha Riesgo Acción recomendada
MD5 sin salt CRÍTICO Tratar como texto plano. Reset inmediato + notificación a usuarios
MD5 con salt ALTO Crackeable con tiempo. Reset + notificación urgente
SHA1 sin salt ALTO Asumir comprometidas. Reset inmediato
SHA256 con salt MEDIO Reset preventivo. Monitorear credential stuffing
bcrypt cost ≥ 12 BAJO Reset como buena práctica. Riesgo real bajo para contraseñas fuertes
Argon2id MUY BAJO Reset recomendado. Cracking es computacionalmente inviable
